# AgentSystem UI/UX 设计规格

版本：0.1  
状态：设计基线  
关联需求：[产品需求文档](product-requirements.md)  
视觉规范：[统一设计系统](../design-system/MASTER.md)

## 1. 设计结论

新版界面采用“任务中心 + 管理控制面”的双层结构：

- 日常研发工作围绕任务工作台展开，项目上下文、Agent 状态、对话、审批、变更、测试和 Trace 保持在同一页面。
- Agent、模型、Provider、凭据、策略和集成属于管理控制面，不与任务执行入口混在同一层级。
- 运行状态以时间线和事件证据为主，拓扑画布作为辅助视图，不承担唯一导航职责。
- “模拟”和“真实”是环境级信任信号，必须持续显示，不能只藏在模型配置详情里。

## 2. 当前界面诊断

基于当前本地界面、DOM 结构和代码实现，主要问题如下：

| 问题 | 用户影响 | 设计调整 |
| --- | --- | --- |
| 8 个一级导航以技术模块命名 | 完成任务时在任务、Agent、工作区、模型和日志之间反复跳转 | 一级导航按用户目标合并为工作台、任务、项目、Agent Studio、运营与治理 |
| 任务创建表单占据首屏 | 任务看板、详情、审批和对话被推到下方 | 新建任务改为抽屉式分步流程；任务页默认展示列表和当前工作 |
| 总览使用大面积拓扑画布 | 空任务时信息稀疏，真实阻塞和行动项不突出 | 工作台优先显示待办、运行中任务、失败和待审批；拓扑作为任务详情视图 |
| Agent 列表与检查器垂直堆叠 | 在常见桌面高度下无法同时看到 Agent 和详情 | 桌面使用列表 + 详情双栏，任务工作台使用可固定右侧检查器 |
| 顶部始终显示相同快捷入口 | 与当前页面动作重复，占用水平空间 | 顶部保留全局新建、搜索、环境、审批和用户入口；页面动作放入页头 |
| 状态百分比由固定映射推算 | 容易被理解为真实进度 | 未提供真实进度时使用阶段状态和不确定进度，不显示伪精确百分比 |
| 模型矩阵与编辑器同页长表单 | 配置比较、验证、版本发布路径不清晰 | Agent Studio 中按配置版本编辑，Provider 与凭据独立管理 |
| 工作区文件为平铺列表 | 代码定位效率低，缺少 Git 与索引信息 | 使用可搜索树、面包屑、文件预览和项目健康摘要 |
| 原始状态值和中英文术语混用 | 非技术审批人难以理解 | 用户文案本地化，原始值只在技术详情或复制区显示 |
| 事件输出缺少等级与关联对象 | 难以从日志判断异常和下一步 | 事件流增加级别、Agent、Step、耗时、关联产物和快速筛选 |

## 3. 体验原则

### 3.1 任务优先

用户进入系统后首先看到需要处理的任务，而不是系统架构。任何任务的主要动作都能在任务工作台内完成。

### 3.2 证据优先

审批、失败和安全告警必须展示证据：涉及文件、命令、模型、测试结果、策略规则和影响范围。抽象状态不能替代证据。

### 3.3 信任可见

真实/模拟、内网/外部、凭据状态、工具权限和数据分级始终可识别。用户不需要打开日志才能确认系统是否会产生外部副作用。

### 3.4 渐进披露

默认显示结论、风险和下一步；输入输出、原始 prompt、token、命令和完整日志按需展开。运维细节不淹没日常研发任务。

### 3.5 上下文不丢失

从任务切换到 Agent、文件、产物或 Trace 时，保留当前任务、筛选条件、滚动位置和未发送输入。

### 3.6 状态真实

不展示推算出的伪精确进度。只有工作流提供可计算总量时才显示百分比；否则使用当前阶段、已完成步骤和持续时间。

## 4. 信息架构

### 4.1 一级导航

| 导航 | 面向对象 | 包含内容 |
| --- | --- | --- |
| 工作台 | 所有用户 | 我的待办、运行中任务、待审批、失败、最近项目、系统告警 |
| 任务 | 开发者、审批人 | 任务列表、保存视图、任务工作台、审批、产物、对话 |
| 项目 | 开发者、团队管理员 | 本地项目、GitHub 仓库、索引、分支、项目策略 |
| Agent Studio | 团队管理员、平台管理员 | Agent 配置、模型路由、工具、handoff、版本、eval |
| 运营与治理 | 运维、安全、审计 | 运行监控、Trace、成本、审计、策略、审批中心 |
| 系统设置 | 平台管理员 | Provider、凭据引用、集成、成员、语言、主题、通知 |

权限不足的入口不直接消失：若入口对角色有教育价值，展示只读说明和申请权限路径；纯管理员入口可在普通用户侧隐藏。

### 4.2 路由建议

```text

/governance/audit

/settings/credentials
/settings/integrations

```

所有关键页面均可深链。任务详情 URL 保存选中的 Run 和主标签；临时抽屉状态使用查询参数或路由状态管理。

## 5. 全局应用框架

### 5.1 顶部栏

从左到右：

1. 产品标识和当前租户。
2. 当前项目切换器；无项目时显示“选择项目”。
3. 全局搜索/命令入口，可搜索任务、项目、Agent 和 Trace。
4. 环境信任标识：`模拟`、`内网真实调用` 或 `外部模型受控调用`。
5. 待审批入口与数量。
6. 系统健康、通知、帮助和用户菜单。

环境信任标识使用文本、图标和语义色，不只依赖颜色。点击后解释可能产生的数据流和副作用。

### 5.2 左侧导航

- 展开宽度 232 px，收起宽度 64 px。
- 使用 Lucide 图标加文本；收起后保留 tooltip 和可访问名称。
- 当前项同时使用背景、左侧指示条和字重表达。
- “新建任务”是导航顶部的单一主要动作。
- 系统设置与退出位于底部，与业务导航分组。

### 5.3 全局反馈

- 小于 300 ms 的操作不显示加载动画。
- 300 ms 至 1 s 使用按钮内进度或局部骨架。
- 大于 1 s 使用骨架并显示当前阶段；长任务进入后台，不阻塞页面。
- Toast 只确认已完成的小动作；错误显示在相关区域并提供恢复动作。
- 危险操作使用确认对话框；包含影响范围和不可逆说明。

## 6. 核心页面规格

### 6.1 工作台

目标：在 10 秒内回答“我现在要处理什么”。

首屏顺序：

1. 待处理条：待审批、需要输入、失败、预算告警。
2. 运行中任务表：任务、项目、当前阶段、活跃 Agent、持续时间、阻塞。
3. 我的任务：最近创建、关注或负责的任务。
4. 系统健康摘要：仅异常时提升到首屏。

空状态不展示空拓扑。无任务时提供“选择项目”和“新建任务”两个明确入口，并展示一条最短上手路径。

### 6.2 任务列表

目标：快速定位、比较和批量管理任务。

- 默认使用紧凑数据表，不使用看板作为唯一视图。
- 列：状态、任务标题、项目/仓库、当前阶段、负责人、优先级、等待时间、更新时间、成本。
- 支持状态、项目、创建人、Agent、风险、时间和标签筛选。
- 支持保存视图：我的运行中、待我审批、失败待处理、最近完成。
- 看板作为可选视图，仅按状态分组；列内使用虚拟列表。
- 行点击进入任务工作台；行尾菜单提供关注、复制链接、取消和重新运行。

### 6.3 新建任务

使用右侧抽屉或独立页面，分为三步：

1. 选择来源：本地项目、已连接仓库或 GitHub Issue。
2. 描述任务：自然语言需求、附件、Issue、目标分支。
3. 执行策略：工作流模板、审批策略、优先级、预算、真实/模拟模式。

设计规则：

- 保持字段标签常驻，placeholder 只提供示例。
- 默认值来自项目策略，用户只处理例外配置。
- 提交前展示“将访问的项目、将使用的模型范围、可能产生的副作用”。
- 表单自动保存草稿；关闭有未保存修改时确认。
- 创建成功后直接进入任务工作台，不停留在列表。

### 6.4 任务工作台

目标：成为单个任务的唯一操作中心。

#### 顶部任务条

- 任务标题、状态、优先级。
- 项目、基础分支、任务分支、Issue/PR 链接。
- 当前 Run、Trace、创建人、持续时间、成本。
- 主要动作随状态变化：开始、审批、重试、继续、取消、打开 PR。
- 菜单包含复制链接、导出报告、复制为新任务。

#### 中央工作区

使用标签页承载不同证据：

| 标签 | 内容 |
| --- | --- |
| 进度 | Step 时间线、当前阻塞、handoff、运行摘要 |
| 变更 | 文件树、diff、补丁版本、风险文件标记 |
| 测试 | 命令、结果、失败用例、修复轮次、报告 |
| 安全 | secret、依赖、权限、外联、Prompt Injection 检查 |
| 审查 | Review Agent 发现、严重级别、解决状态、人工评论 |
| 产物 | 计划、上下文包、补丁、日志、报告、PR 描述 |
| Trace | 模型、工具、审批和系统事件的完整时间线 |

“协作拓扑”位于进度标签内，可切换为“时间线”。默认使用时间线，因为它更适合判断执行顺序和阻塞。

#### 右侧 Agent 控制台

宽度 360 至 420 px，可固定或收起。包含：

1. Agent roster：名称、状态、当前动作和模型；按工作流顺序排列。
2. Agent Inspector：目标、输入摘要、输出摘要、模型、工具、handoff 和错误。
3. 协作对话：选择 Agent 后发送消息；消息明确标记“建议”“已采纳”“触发重新规划”。

对话不能直接绕过工作流。若消息会改变计划、补丁范围或危险动作，系统先展示建议，再由 Orchestrator 创建新 Step 或审批。

#### 底部事件控制台

- 默认高度 240 px，可收起、拉伸和全屏。
- 标签：全部、Agent、模型、工具、审批、系统、错误。
- 每行包含时间、级别、Actor、事件、耗时和简短结果。
- 点击事件在右侧打开结构化详情，不在列表中直接展示大段 JSON。
- 支持暂停自动滚动、关键字过滤、复制事件 ID 和跳转相关产物。

### 6.5 项目

目标：明确“系统能访问哪些代码，以及代码是否准备好被 Agent 使用”。

项目列表展示来源、仓库、本地路径、默认分支、索引状态、策略和最近任务。

项目详情包括：

- 概览：Git 状态、语言、依赖、索引健康、风险提示。
- 文件：可搜索树、面包屑、文件预览、符号大纲。
- 上下文：索引版本、排除规则、敏感级别、重建索引。
- 工作流：默认模板、测试命令、审批策略和风险路径。
- 访问：成员、GitHub App 权限、短期 token 状态。

选择本地目录必须使用系统目录选择器。路径只读展示；更换目录是显式动作。

### 6.6 Agent Studio

目标：配置和发布可治理的 Agent，而不要求管理员直接修改环境变量或源代码。

左侧为 Agent 列表和状态；中央为配置；右侧为版本和验证结果。配置标签：

- 概览：名称、职责、Owner、启用状态。
- 指令：系统指令、输入输出 schema、上下文策略。
- 模型：主路由、回退、credential reference、预算、超时。
- 工具：allowlist、路径、网络、资源限制和危险级别。
- Handoff：可交接 Agent、条件、上下文映射。
- Evals：评估集、最近得分、回归差异。
- 版本：草稿、发布、归档、回滚和变更记录。

保存只产生草稿；发布前必须通过 schema 检查、凭据可用性检查和最小 eval 门禁。

### 6.7 运营与治理

运营视图：运行量、成功率、P95 时长、模型延迟、成本、失败原因和队列。图表必须提供表格替代与筛选。

审批中心：按风险、项目、类型和等待时间排列；详情使用标准审批证据模板。

审计：结构化表格、资源深链、导出、保留策略；不把原始 JSON 作为默认阅读方式。

策略：策略列表、作用域、版本、命中次数、模拟评估和发布记录。

### 6.8 系统设置

按职责分组：

- Provider 与模型网关。
- Credential References。
- GitHub Enterprise 与其他集成。
- 成员、角色和租户。
- 通知与保留策略。
- 个人偏好：语言、主题、时区、密度和减少动画。

语言和主题属于个人偏好，不与平台级安全策略放在同一面板。

## 7. 审批交互规范

审批卡固定包含：

- 审批类型和风险等级。
- 触发策略与原因。
- 即将发生的动作。
- 影响范围：项目、分支、文件、外部目标。
- 证据：计划、diff、测试、安全与审查摘要。
- 请求时间、有效期、请求者和所需角色。

动作顺序：`批准`、`要求修改`、`拒绝`。拒绝使用危险色；批准是当前页面唯一主要按钮。

提交后显示不可编辑决策摘要。并发冲突时提示“该审批已由某人处理”，刷新状态且不重复提交。

## 8. 状态与文案

### 8.1 状态命名

| 数据值 | 中文 | 英文 |
| --- | --- | --- |
| created | 已创建 | Created |
| queued | 排队中 | Queued |
| running | 运行中 | Running |
| awaiting_approval | 待审批 | Awaiting approval |
| input_required | 需要输入 | Input required |
| succeeded/completed | 已完成 | Completed |
| failed | 失败 | Failed |
| canceled | 已取消 | Canceled |
| skipped | 已跳过 | Skipped |
| simulated | 模拟 | Simulated |

原始状态值可在技术详情中复制，但主要界面只使用本地化名称。

### 8.2 错误文案结构

错误文案必须回答：发生了什么、可能原因、用户能做什么、相关 ID 是什么。

示例：

> 测试步骤超时，任务已安全停止。项目测试命令运行超过 30 分钟。你可以查看日志、调整项目超时策略，或从测试步骤重新运行。Trace: `trace_xxx`。

避免只显示 `Internal Server Error`、`Failed` 或原始异常堆栈。

### 8.3 空状态

- 没有任务：说明价值并提供新建任务。
- 没有待审批：显示“当前没有需要你处理的审批”，不显示空白容器。
- 没有事件：解释事件会在运行开始后出现。
- 无权限：说明缺少什么权限、由谁授权和可执行的下一步。
- 集成离线：显示影响范围和重试/联系管理员入口。

## 9. 响应式与密度

### 9.1 桌面 >= 1440 px

- 固定左导航 + 中央内容 + 可固定右侧 Agent 控制台。
- 任务工作台三栏同时可见。
- 事件控制台位于底部。

### 9.2 小桌面 1024-1439 px

- 左导航默认收起。
- Agent 控制台作为右侧抽屉；打开时不覆盖关键审批按钮。
- 数据表按优先级隐藏次要列，并提供列设置。

### 9.3 平板 768-1023 px

- 顶部导航或抽屉导航。
- 任务详情、Agent 和事件使用标签页切换。
- 支持查看、评论和审批；复杂 Agent 配置提示使用桌面端。

### 9.4 手机 < 768 px

- 只保留工作台、任务、审批、通知和个人设置。
- 代码 diff 使用文件级切换，避免横向页面滚动。
- 不支持复杂工作流画布编辑和 Agent 发布。

## 10. 键盘与可访问性

- 首个焦点为“跳到主内容”。
- `Cmd/Ctrl + K` 打开全局搜索；快捷键可在设置中关闭。
- `g` 后跟导航键只在焦点不位于输入控件时生效。
- 所有 icon-only 按钮有 `aria-label` 和 tooltip。
- Drawer/Modal 打开后焦点进入容器，关闭后返回触发元素。
- 动态事件使用 `aria-live="polite"`，错误使用 `role="alert"`，不抢占输入焦点。
- 工作流图提供等价时间线；图表提供数据表和文字摘要。
- 焦点环至少 2 px；正文对比度 >= 4.5:1；UI 图形 >= 3:1。
- 动画尊重 `prefers-reduced-motion`；不使用闪烁或纯装饰持续动画。

## 11. 国际化

- 文案使用语义 key，不在组件中拼接中文和英文片段。
- 日期、时间、数字、成本按 locale 和用户时区格式化。
- 布局为英文较长标签预留 30% 空间；按钮允许换行或使用更短规范文案。
- 代码、模型名、provider、路径和 ID 不翻译。
- 中英文术语表由产品维护，避免 Agent/智能体、Run/运行等词在同一层级随机混用。

## 12. 数据可视化

- 趋势使用折线图；Agent/模型比较使用水平条形图；状态分布优先使用堆叠条形图。
- 超过 5 类不使用饼图。
- 所有图表有单位、图例、tooltip、时间范围和数据表替代。
- 成功/失败不能只用红绿区分；使用图标、文字和形状。
- 1000 个以上事件先聚合，支持按时间段下钻。

## 13. 设计验收清单

- 核心任务可在任务工作台完成，无需切换到 Agent、模型或日志一级页面。
- 首屏可见当前状态、阻塞、下一动作和信任模式。
- 没有伪精确进度或无法解释的百分比。
- 真实调用、模拟调用和外部数据流在全局与调用级均可识别。
- 所有异步操作有加载、成功、失败和重试状态。
- 危险操作展示影响范围并要求确认或审批。
- 375、768、1024、1440 和 1920 px 下无不可控横向滚动和内容遮挡。
- 浅色、深色、高对比度、200% 缩放和减少动画均通过检查。
- 键盘可以完成创建任务、查看结果、发送消息和审批流程。
- 中文和英文界面均无截断、重叠和未翻译文案。
